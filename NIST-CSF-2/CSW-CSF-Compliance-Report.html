Cisco Secure Workload — NIST CSF 2.0 Compliance Mapping

How to read this report

This report describes how Cisco Secure Workload (CSW) supports an organisation’s posture against the NIST Cybersecurity Framework 2.0 (released February 2024). CSF 2.0 is not a control catalogue — it’s an outcomes framework that wraps control catalogues like NIST 800-53, ISO 27001, CIS Controls, and others through Informative References. The right way to read this report is as the “where does CSW evidence land in the CSF 2.0 narrative” view.

What changed in CSF 2.0. A sixth Function — Govern (GV) — was added above Identify, Protect, Detect, Respond, Recover. Govern formalises expectations around organisational context, risk strategy, roles and responsibilities, policy, oversight, and cybersecurity supply chain risk management. Several Categories and Subcategories within the existing five Functions were reorganised and refined.

Companion document. The accompanying CSF 2.0 Technical Runbook walks through CSW deployment per Profile / Tier, and the Subcategory-by-Subcategory implementation steps. Read this report for the audit-narrative view; read the runbook for the deployment view.

1. Executive Summary

CSF 2.0 organises cybersecurity outcomes across six Functions, 22 Categories, and 106 Subcategories. CSW is the workload-tier evidence engine for the technical Subcategories under Identify (asset management, risk assessment), Protect (platform security, infrastructure resilience), Detect (continuous monitoring, adverse event analysis), and Respond (analysis, mitigation). Govern and Recover are organisational/process Functions where CSW supplies inputs but is not the primary control.

What CSW changes for a CSF 2.0 programme:

  • GV.OV / GV.SC — Oversight and Supply Chain. Standing quarterly evidence pack to the management body covering the six CSF Functions with technical detail (workload inventory drift, vulnerability backlog, segmentation diff, detection trends, incident summary, vendor-egress drift). Governance gets ground- truth, not green-yellow-red dashboards.
  • ID.AM / ID.RA — Asset Management and Risk Assessment. Continuous workload inventory + ADM dependency graph + per- workload CVE inventory with reachability-aware prioritisation.
  • PR.IR / PR.PS — Infrastructure Resilience and Platform Security. Workload-level deny-by-default segmentation enforcement (PR.IR-01); daily configuration baseline with drift detection (PR.PS-01); software lifecycle including vendor-EOL tagging (PR.PS-02); SIEM forwarding (PR.PS-04).
  • DE.CM / DE.AE — Continuous Monitoring and Adverse Event Analysis. The densest CSW alignment in CSF. Per-workload process + flow telemetry; behavioural rules; conversation-graph anomaly detection; SIEM correlation.
  • RS.AN / RS.MI — Incident Analysis and Mitigation. Six- artefact reconstruction bundle for every declared incident; quarantine policy templates pre-built per scope.

What CSW does not change. CSW does not address physical/ environmental Subcategories (PR.IR-02, DE.CM-02), backup/ recovery Subcategories (RC.RP-02/03), or training/HR-style Subcategories. Identity is upstream (PR.AA is supporting; CSW enforces workload-side outcomes of upstream identity decisions).

2. Compliance Posture Summary

Function Coverage CSW contribution
Govern (GV) Supporting Quarterly evidence pack; vendor-egress observation feeding GV.SC
Identify (ID) Direct ID.AM: continuous workload + flow + software inventory. ID.RA: continuous CVE + reachability + drift evidence
Protect (PR) Direct PR.IR: segmentation enforcement. PR.PS: baseline + drift + SIEM + behavioural rules. PR.AA / PR.DS: supporting (workload-side enforcement of upstream identity, plaintext DENY)
Detect (DE) Direct DE.CM: workload telemetry + behavioural rules + anomaly detection. DE.AE: forensic timeline + SIEM correlation + threat intel
Respond (RS) Direct RS.MA / RS.AN: six-artefact dossier + severity routing + magnitude estimate. RS.MI: quarantine policy templates
Recover (RC) Supporting Pre-incident segmentation export available for re-application; post-recovery diff = evidence of new normal

Posture takeaway. Direct on Identify, Protect, Detect, Respond (the technical core of the framework). Supporting on Govern (where CSW supplies evidence the management body uses) and on Recover (where CSW supplies pre/post-incident state diffs but is not the recovery executor itself). This is consistent with CSW’s positioning as a workload-protection platform: it is the evidence engine for the operational Functions and the input source for the governance Function.

RC.RP-02 / RC.RP-03 Backup / DR team Backup product + integrity testing

CSW’s role in a complete CSF 2.0 programme is the workload-tier evidence engine for the technical Functions (ID, PR, DE, RS) and the input source for the governance Function (GV) and the operational evidence layer for Recover (RC). Identity, physical, backup, training, and crypto-at-rest are explicitly upstream.

5. Suggested Next Steps

For organisations adopting or maturing CSF 2.0 posture with CSW:

  1. Confirm your Profile. CSF 2.0 expects Current Profile, Target Profile, and progress measurement between them. Get security leadership to sign off the Profile before scoping CSW labels and scopes — CSW evidence aggregates per Subcategory and per scope, so the Profile structure determines the report structure.
  1. Pilot on ID.AM, PR.IR, DE.CM. These are the densest CSW alignment in CSF and produce the fastest visible wins. Inventory and ADM are the foundation; PR.IR is where the segmentation work lives; DE.CM is where the daily detection work lives.
  2. Build the GV evidence pack early. CSF 2.0’s new Govern Function expects management-body engagement. The quarterly evidence pack (workload inventory drift, vulnerability backlog, segmentation diff, detection trends, incident summary, vendor- egress drift) is the artefact governance asks for once and reuses every cycle.
  3. Wire CSW telemetry into the SIEM you already have. CSF DE.CM and DE.AE rely on correlation across data sources; CSW is one source among many. The uplift comes from richer workload-tier data, not from replacing the SIEM.
  4. Standardise the RS.AN-07 incident dossier template. The six-artefact bundle is the artefact RS subcategories converge on; build it once and reuse.
  5. Plan SME review. This document is a v1 mapping. Before being relied upon in a formal compliance engagement, route to a CSF specialist for currency (CSF Quick-Start Guides and Implementation Examples evolve) and to a current Cisco Secure Workload SE for product-capability accuracy.

6. Disclaimer

This report and its companion runbook describe how Cisco Secure Workload capabilities can support a NIST Cybersecurity Framework 2.0 programme. They are not legal or compliance advice and do not constitute a finding of compliance.

Specifically:

  • NIST publishes the authoritative CSF 2.0 text and Subcategory list at https://www.nist.gov/cyberframework. Always validate against the current effective version of CSF, including any released Quick-Start Guides, Profiles, Implementation Examples, and Informative References mappings.
  • CSF 2.0 is an outcomes framework and does not prescribe specific controls. Mapping a CSW capability to a Subcategory does not by itself satisfy the Subcategory — the organisation’s Profile, Tier, and accompanying control set determine the depth of work expected.
  • CSW does not address physical/environmental Subcategories, backup/ recovery Subcategories, or training/HR-style Subcategories. Reference the appropriate parts of your programme for those.

This document should receive subject-matter-expert review for both CSF 2.0 currency and current Cisco product capability before being used in a customer engagement.
